# 暂停与恢复监听设计

## 目标

允许用户临时暂停 ThreadBeacon 每两秒执行一次的自动刷新，在稳定查看任务状态或暂时不需要更新时保持当前界面不变，并能一键恢复监听。该功能只暂停 ThreadBeacon 的本地只读刷新，不影响 Codex 任务本身。

## 对齐基准

行为与 macOS 版本 `965d31e`、`b2840a1`、`d78c18f` 和 `737409b` 对齐：

- 标题栏在图钉和手动刷新之间增加监听控制按钮。
- 正在监听时显示暂停图标，标签为“暂停监听”。
- 已暂停时显示播放图标和强调色，标签为“恢复监听”。
- 暂停后停止后续自动刷新；已开始的只读刷新允许完成。
- 恢复时立即刷新一次，然后继续每两秒自动刷新。
- 暂停期间手动刷新仍可使用。
- 暂停状态不持久化，重新启动 App 后默认恢复监听。

## Windows 交互与布局

标题栏按钮顺序固定为：图钉、暂停/恢复、手动刷新。三个按钮均保持 32 x 32 像素，按钮间距为 8 像素，使用 `Segoe Fluent Icons`：

- active：`Pause` 图标，次要文字色，Tooltip 与无障碍名称为“暂停监听”。
- paused：`Play` 图标，Windows 强调蓝，Tooltip 与无障碍名称为“恢复监听”。

Footer 继续使用现有左右布局：

- active：左侧显示数据源/任务摘要，右侧显示最后更新时间。
- paused 且已有更新时间：左侧显示“监听已暂停 · 上次更新”，右侧显示时间。
- paused 且尚未更新：左侧显示“监听已暂停 · 尚未更新”，右侧为空。
- 数据源缺失、繁忙、不兼容或刷新失败时，错误信息优先于暂停提示，避免隐藏真实问题。

## 架构

`ThreadBeacon.App` 新增纯 UI 状态 `MonitoringState`：

- 默认 active，不读取或写入本机设置。
- 暴露 `IsPaused`、`ShouldAutoRefresh` 和 `ToggleCommand`。
- 切换时通知 `MainWindowViewModel` 和 WPF 绑定更新。

`MainWindowViewModel` 持有 `MonitoringState`，并根据最近一次加载结果计算 Footer 文本。暂停不会清空任务、状态或 Token 数据，手动 `RefreshCommand` 不受暂停状态限制。

`MainWindow` 继续拥有 `DispatcherTimer`：

- 初次加载时立即刷新并启动计时器。
- 监听状态切到 paused 时停止计时器。
- 切回 active 时立即调用一次 `RefreshAsync()`，完成后重新启动计时器。
- `RefreshAsync()` 现有信号量继续防止恢复、手动刷新和计时器刷新并发执行。

`ThreadBeacon.Core` 不增加 WPF 或监听生命周期依赖；暂停是 Windows App 的进程级交互状态，不属于 Codex 数据模型。

## 错误与边界

- 暂停发生在刷新进行中时不取消该刷新，避免中途终止 SQLite/JSONL 读取。
- 恢复期间若已有刷新，现有并发门会安全返回；计时器随后继续工作。
- App 关闭时停止计时器并取消状态订阅。
- 暂停期间任务列表保持最后一次成功快照，不改写 Codex 数据。

## 测试与验收

1. 单元测试证明监听状态默认 active，并能在 paused/active 间切换。
2. 单元测试证明暂停 Footer 保留最后更新时间，错误状态优先。
3. 全部 Core 与 App 测试通过，Release 构建无警告和错误。
4. 真实 App 中暂停至少五秒，更新时间不自动变化。
5. 暂停期间手动刷新，更新时间只前进一次，随后继续保持。
6. 恢复后立即刷新，并在至少两个后续两秒周期继续更新时间。
7. 重启 App 后恢复 active，按钮显示“暂停监听”。
8. 窄至 480 像素时三个标题栏按钮不与标题或列表重叠。

## 本阶段不做

- 自定义刷新间隔。
- 持久化暂停状态。
- 快捷键、托盘或任务栏菜单控制。
- 按任务分别暂停。
- 暂停或终止 Codex 任务。
- 完成提示音、Subagent 展开、429/503、任务置顶/忽略等后续功能。
