# Windows 主题设置设计

## 背景

macOS 版本已支持 `System`、`Light` 和 `Dark` 三种主题，并在主窗口、设置窗口和弹窗中即时生效。Windows 版本当前在 `App.xaml` 中只有一套浅色画刷，界面大量使用 `StaticResource`，因此不能通过简单替换一两个颜色值实现完整主题切换。

## 目标

- 在设置窗口的“通用”页增加主题选择：跟随系统、浅色、深色。
- 默认值为跟随系统，非法或缺失配置回退到跟随系统。
- 主题选择保存到现有显示设置 JSON，不新增网络或外部服务依赖。
- 切换后主窗口、Settings、Token/健康弹窗和菜单立即更新。
- 状态灯、健康等级和错误等级继续保留独立的语义颜色，并在两种背景下保持可读。
- Windows 系统主题变化时，跟随系统模式自动更新。

## 非目标

- 本阶段不加入自定义颜色、主题编辑器或高对比度专用主题。
- 不改变任务数据读取、刷新策略、窗口置顶、托盘或声音功能。
- 不翻译用户任务标题、Subagent 标题或其他 Codex 原始内容。

## 方案

### 主题模型与持久化

在 `ThreadBeacon.App` 增加 `AppTheme` 枚举：

- `System`
- `Light`
- `Dark`

`DisplaySettings` 增加 `Theme` 字段，使用显式 JSON 转换或稳定字符串值保存。读取未知值时使用 `System`。现有旧版 JSON 缺少字段时也必须得到 `System`，不能影响其他显示设置。

`DisplaySettingsViewModel` 暴露主题选项和双向绑定属性。主题变更立即保存，并通过 `AppThemeState` 广播给所有窗口。

### 资源与刷新

将当前 `App.xaml` 的颜色资源拆分为：

- `Resources/Theme.Light.xaml`
- `Resources/Theme.Dark.xaml`

两套字典提供相同的资源键：窗口背景、表面、主文字、次文字、边框和必要的控件状态颜色。所有用户界面颜色引用迁移为 `DynamicResource`，避免已创建的窗口继续使用旧画刷。

`AppThemeState` 负责：

1. 保存用户偏好和当前解析后的主题。
2. 将 `System` 解析为当前 Windows 应用主题。
3. 移除旧主题字典并加入新字典。
4. 发出变更通知，使资源、设置选择和辅助文本保持同步。

应用启动时先加载显示设置，再应用主题资源，然后创建主窗口。Settings 窗口和所有 Popup 继续使用应用级资源，避免复制一套主题状态。

### 系统主题检测

`System` 模式读取 Windows 个性化设置中的应用浅色/深色偏好，并监听用户偏好变化。系统检测失败时使用浅色作为保守的可读回退，但不覆盖用户保存的 `System` 偏好。切换到明确的浅色或深色后，不再响应系统主题变化。

### 颜色语义

状态灯颜色不跟随主题简单反转，而是继续表达运行、完成、警告和错误。需要为深色背景单独验证画刷明度和文本对比度；健康盾牌、状态圆点和错误文字不能只依赖颜色区分。

## 测试策略

- `AppTheme`：枚举顺序、默认值、字符串转换和非法值回退。
- `DisplaySettings`：旧 JSON 无主题字段、主题往返保存、其他显示设置不被覆盖。
- `AppThemeState`：System/Light/Dark 解析、资源字典切换和变更通知。
- XAML 静态检查：浅色和深色字典提供相同资源键；界面颜色使用 `DynamicResource`，不残留关键 `StaticResource`。
- ViewModel：主题选项显示名称随界面语言变化，主题切换立即保存。
- 电脑功能验收：打开 Settings 切换三种主题，主窗口、设置窗口、Token 详情和健康弹窗同时更新；关闭并重新打开程序后主题保持。

## 验收标准

在本机 Windows 11 上：

1. 默认安装启动显示跟随系统主题。
2. 切换浅色或深色后，主窗口和已打开的 Settings/Popup 在一次操作内更新，无需重启。
3. 切换回跟随系统后，系统主题变化可以驱动界面更新。
4. 主题切换不清空刷新间隔、最大任务数或语言选择。
5. 两套主题下任务状态、健康盾牌、Token 详情和菜单文字均无明显重叠或不可读。
6. 全部自动化测试通过，安全检查无敏感信息，提交并推送后重新安装最新 EXE。
